# YUWA ロードマップ・進め方

> **対象読者**: エンジニア（実装担当）、青木（PO）、運営チーム
> **目的**: 「どう進めるか」を共有し、不要なすれ違いをなくす
> **作成日**: 2026-05-25

---

## 1. フェーズ概要

```
Phase 0: 準備             ← 完了
   ├ 要件定義 v0.1
   ├ ブランド決定（YUWA）
   └ リポ・ドキュメント整備

Phase 1: 環境構築          ← ここから（合流したエンジニアの最初のタスク）
   ├ 要件 v0.2 ドラフト（PO 主導、出欠/スケジュール/名簿の追加）
   ├ Next.js プロジェクト初期化
   ├ Supabase プロジェクト作成
   └ Vercel preview 環境のセットアップ

Phase 2: コア機能実装
   ├ 認証フロー（招待コード → Google ログイン）
   ├ 意見投稿 + モデレーションキュー
   └ イベント掲示板

Phase 3: 内部ドッグフード
   ├ 運営チーム 3-4 名で先に使ってみる
   └ バグ修正・UI 微調整

Phase 4: ローンチ
   ├ 利用規約 正式版
   ├ 部メンバーへの招待コード配布
   └ 使い方説明会

Phase 5: v0.2 機能拡張
   ├ 出欠管理
   ├ スケジュール管理
   └ 名簿管理
```

各フェーズの想定期間とマイルストーンは「6. マイルストーン」参照。

## 2. 体制と役割

| 役割 | 担当 | 主な責任 | コミット時間（目安） |
|---|---|---|---|
| **PO（プロダクトオーナー）** | 青木 | 要件・規約・運営判断・部内調整・PR レビュー（UI/UX 観点） | 週 5-10 時間 |
| **Eng（エンジニア）** | 部員ボランティア | 実装全般、技術判断、PR 作成 | 本人ペース |
| **運営チーム** | 部の運営メンバー 3-4 名 | ローンチ後のモデレーション | ローンチ後 |

### 重要な配慮：「ボランティア前提」

エンジニアはボランティアで参加してくれている。これは仕事ではなく、本人の善意と時間の提供。

**避けるべきこと**:
- 「いつまでにやって」と一方的な納期要求
- 「もっとこうして」と過剰な手戻り
- 緊急ではない issue で深夜に連絡
- レビューを必須化して負担増

**心がけること**:
- 「ありがとう」を毎回伝える
- 進捗が遅れても催促しない、相談ベース
- PO の青木が「これは外していい？」と機能削減判断
- 詰まったら一緒に考える、抱え込ませない

## 3. コミュニケーション

### 日常のやり取り
- **Slack または LINE**（青木とエンジニアの間で決定）
- 即レス不要、24-48 時間で返信があれば OK

### 仕様議論
- 簡単な確認 → Slack/LINE
- 設計判断や複雑な議論 → GitHub Issue または 直接会話（部会の前後など）

### 進捗共有
- 週 1 回、青木が「今週の進捗どんな感じ？」と軽く聞く
- 詰まりや迷いがあれば共有
- 明確な「報告義務」は設けない

## 4. レビュー・マージ方針

### PR ベース開発
- main ブランチへの直接 push 禁止
- すべての変更は PR を切る

### レビューの考え方
- **PO の青木がレビュー** — 非エンジニアでも UI/UX、要件との合致、規約抵触の有無は判断できる
- 技術的な細かいレビューは無理に求めない
- **セルフマージ可** — 軽微な変更や急ぎの修正はエンジニア自身がマージしても OK
- ただし以下は事前に PO 相談:
  - DB スキーマ変更
  - 認証フローの変更
  - 規約・個人情報の扱いに関わる変更
  - 大きな依存追加（Stripe など外部サービス連携）

### main の状態
- main は常に動く状態を維持
- 壊れたまま放置しない（修正 PR を急ぐ）

### コミット規約
- Conventional Commits 推奨（`feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `test:` 等）
- 厳格には強制しない、明確な変更ログがあれば OK

## 5. 推奨技術スタック

要件定義 v0.1 で合意した技術選定。エンジニアの判断で変更可能だが、変更時は PO と相談。

| 領域 | 技術 | 理由 |
|---|---|---|
| フロントエンド | **Next.js 15+ (App Router)** | エコシステム成熟、Vercel との親和性、SSR/SSG 両対応 |
| 言語 | **TypeScript** | 型安全性、IDE 補完、引き継ぎ容易 |
| スタイリング | **Tailwind CSS** | 開発速度、デザイン一貫性、コンポーネント設計と相性良い |
| UI ライブラリ | **shadcn/ui**（推奨） | アクセシビリティ、Tailwind ネイティブ、コードコピー方式で依存が浅い |
| DB / 認証 / API | **Supabase**（東京リージョン） | PostgreSQL + Auth + Storage がワンセット、無料枠が十分 |
| ホスティング | **Vercel**（Hobby plan） | Next.js 公式、preview deploy 自動、無料枠で十分 |
| パッケージマネージャ | **pnpm** | 高速、ディスク効率、Vercel もサポート |
| ビルドツール | **Turbopack**（Next.js 標準） | デフォルトで有効 |

### 注意点
- **pnpm 11 のビルドスクリプト承認**: `sharp` や `unrs-resolver` で承認が要る。`pnpm approve-builds` で対応
- **Next.js のバージョン**: 過去に試した時点で 16.x がリリースされていた。エンジニア合流時の最新版を使う
- **shadcn/ui のインストール**: `pnpm dlx shadcn@latest init`

## 7. エンジニア向け：環境セットアップ手順

### 前提
- Node.js 20 以上（推奨 v24 系）
- Git
- GitHub アカウント
- VS Code または好きなエディタ

### リポを取ってくる
```bash
git clone https://github.com/aokiyuwadayo/yuwa.git
cd yuwa
```

### Phase 1 の最初の作業
1. PO の青木と顔合わせ、要件 v0.1 と事業計画書を一緒に読む
2. v0.2 で追加する機能（出欠 / スケジュール / 名簿）を一緒に整理
3. Next.js プロジェクト初期化:
   ```bash
   pnpm create next-app@latest . \
     --typescript --eslint --tailwind \
     --src-dir --app --turbopack \
     --import-alias "@/*" \
     --use-pnpm --yes
   # ビルドスクリプト承認エラーが出たら:
   pnpm approve-builds
   ```
4. Supabase プロジェクト作成（東京リージョン）→ `.env.local` に接続情報

### 不明点
- 仕様: 青木に Slack/LINE で聞く
- 技術: 自分の判断で進めて OK。詰まったら相談

## 8. リリース後の運用

### モデレーション
- 運営チーム 3-4 名が交代でモデキュー処理
- SLA: 48 時間以内に承認 / 却下
- 詳細は [`requirements-v0.1.md`](requirements-v0.1.md) の §8

### バグ対応
- main が壊れたら最優先で修正
- ユーザー報告は Slack/LINE で受ける（GitHub Issue は内部用）

### 機能追加
- v0.2 機能（出欠 / スケジュール / 名簿）を順次
- メンバーからの要望は意見箱経由で集める

### 卒業・引き継ぎ
- 青木が卒業する前に、後任 PO と引き継ぎセッション
- 主要ドキュメントとリポへの権限委譲
- 技術引き継ぎ書を `docs/handover.md` として残す（卒業前タスク）
