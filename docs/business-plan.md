# YUWA 事業計画書

> **対象読者**: YUWA を一緒に作る人（エンジニア・運営チーム）
> **目的**: 「なぜ作るのか」の共通認識を作る
> **作成日**: 2026-05-25
> **作成者**: 青木

---

## 1. 背景

学生サークルには「Slack / LINE / Google Forms / スプレッドシート」の組み合わせで運営している団体が多いが、これらは個別の用途に最適化されているため、サークル運営の核心的な課題（**心理的安全性のある場・居場所感・自発的な集まり**）を支えるには不十分。

特に YUWA の発祥地である **福岡工業大学（FIT）起業部** では、

- メンバーの「言いづらい意見」が運営に届かない
- イベント参加のハードルが高く、一人で行けない人が多い
- 部の活動情報・名簿・スケジュールが Slack の流れに埋もれる

という課題が顕在化していた。これを **「サークルメンバーが居心地よく集まり、意見を交わし、一緒に動ける場」** として 1 つのアプリで支えるのが YUWA の起点。

## 2. ビジョン

> **学生サークルに「居場所」と「つながり」をテクノロジーで返す。**

サークルは本来、学生にとって最初の「自分が選んだ居場所」であり、最初の「組織運営の経験」でもある。その居場所を、運営者と参加者の両方にとって持続可能で温かいものにする。

## 3. ミッション

サークルメンバーの「言いたいことを安全に言える」「気軽に集まれる」「運営を負担なく続けられる」を支える Web アプリを提供する。

## 4. 解決する課題

| 課題 | 影響 | YUWA の対応 |
|---|---|---|
| メンバーの本音が運営に届かない | 改善サイクルが回らない / 不満の蓄積 | **匿名意見箱**（ハッシュ化匿名 + モデキュー） |
| 一人で行きづらいイベントに行けない | 機会損失 / つながりが育たない | **イベント掲示板**（同行者の可視化） |
| 出欠・スケジュール・名簿が散らばる | 運営者の負担 / 情報の取りこぼし | **サークル管理機能**（v0.2 以降） |
| 運営の判断・指示が見えにくい | 信頼感の低下 | **モデレーションキュー**（透明性のある運営） |

## 5. ターゲットユーザー

### フェーズ 1（MVP・現在）
**福岡工業大学 起業部のメンバー約 40 名（推定）と運営チーム 3-4 名。**

### フェーズ 2（PMF 後）
他大学の **起業系サークル**（イノベーション・ベンチャー・スタートアップ系の名称）。共通の文化があり、横展開しやすい。

### フェーズ 3（さらに先）
学生サークル一般（部活・同好会含む）。ここまで来ると競合（既存サークル管理 SaaS）との比較になる。

## 6. ソリューション

### コア機能（v0.1）
- **匿名意見箱** — ハッシュ化匿名 + 公開前モデレーションキュー方式
- **イベント掲示板** — 「私も行く」表明で同行者を可視化
- **メンバー認証** — 招待コード + Google ログイン
- **モデレーション** — 運営による公開前承認、BAN 可能

### 拡張機能（v0.2 以降）
- **出欠管理** — 部会・イベントへの出欠記録
- **スケジュール管理** — 部の予定の一元化
- **名簿管理** — メンバー情報の本格的な管理

詳細は [`requirements-v0.1.md`](requirements-v0.1.md) を参照。

## 7. 競合と差別化

| 既存ツール | 不足点 | YUWA の優位 |
|---|---|---|
| Slack / LINE | 匿名性なし、モデキュー不可、流れる | 匿名性 + 蓄積される意見箱 |
| Google Forms | 蓄積されるが一覧性なし、UX 弱 | 一覧 + リアクション + モデキュー |
| Discord | 連携が個人依存、サークル特化UI ではない | サークル運営に特化した UX |
| OneCampus 等の大学アプリ | 大学公式・運営が限定的 | サークル単位で自由に運営 |
| 既存サークル管理 SaaS（Univ 等） | 機能てんこ盛り・無料枠制限・UI 古い | 居場所感重視・モダンUI・無料 |

**最大の差別化要素**: 「匿名意見箱 × モデレーション」の組み合わせ。これは「心理的安全性」を技術で担保する設計であり、サークル文化への深い理解がないと作れない。

## 8. ビジネスモデル

**現時点では収益は考えない。** その都度検討する。

- MVP は完全無料運用
- ホスティング費用は Vercel + Supabase の無料枠で月数百円以内に収まる想定 → 青木個人 or 部費で吸収
- 将来的に他大学展開や機能拡張が進んだ際に、ホスティング負担分の有償プランや学校単位のサブスクなどを検討する余地はある（現時点では仮説）

## 9. 体制（2026-05-25 時点）

| 役割 | 担当 | 責任範囲 |
|---|---|---|
| プロダクトオーナー (PO) | **青木** | 要件・規約・運営判断・部内調整・PR レビュー（非エンジニアでも UI/UX 観点で） |
| 開発エンジニア | **部員ボランティア 1 名**（合流予定） | 実装全般、技術判断、Next.js + Supabase 構成 |
| モデレーター（運用フェーズ） | 部の運営チーム 3-4 名 | 投稿の承認 / 却下 / BAN、ローンチ後にアサイン |
| アドバイザー（必要時） | OB ネットワーク | 法務・規約・経営観点での相談 |

**重要な配慮**: エンジニアはボランティアで参加してくれているため、

- 過度な納期プレッシャーをかけない
- レビュー・コミュニケーションはカジュアルに
- 「やってくれてありがとう」を常に持つ

詳細は [`roadmap.md`](roadmap.md) の「コミュニケーション方針」を参照。

## 10. 成功指標 (KPI)

### ローンチ時 (Week 4 目標 = 2026-06-19 頃)
- ✅ FIT 起業部メンバーの **50% 以上**（20 名以上）が招待コードを使ってサインアップ
- ✅ ローンチ後 2 週間で意見投稿が **5 件以上**
- ✅ ローンチ後 2 週間でイベント投稿が **2 件以上**

### 3 ヶ月後（2026-09 頃）
- 月次アクティブメンバー (MAM) **15 名以上** をキープ
- モデキュー処理 SLA 48 時間以内を **90% 達成**
- 1 件以上の「YUWA で出てきた意見がきっかけで部の運営が変わった」事例

### 半年後（2026-11 頃）
- 他大学サークル **1 団体** が試験運用に興味を示す
- v0.2（出欠・スケジュール・名簿管理）リリース完了

## 11. リスクと対策

| リスク | 影響度 | 対策 |
|---|---|---|
| 開発がボランティア依存 → スケジュール遅延 | 高 | マイルストーンに余裕、機能の優先順位を明確化。PO が機能削減判断 |
| 匿名性の運用ミスで信頼失墜 | 高 | 規約での明示 + DB レベルの分離 + モデレーター複数体制で歯止め |
| 利用規約・個人情報保護の不備 | 中 | ローンチ前に運営チームでレビュー、可能なら OB 法務リソースに相談 |
| 学生プロジェクトの継承性（青木卒業後） | 中 | ドキュメント整備、技術スタック選定時に学習コスト考慮、引き継ぎ書を残す |
| ローンチ後に投稿が集まらない | 中 | ローンチ前に「使い方説明会」を部会で実施、運営から初期投稿を出して呼び水に |
| 学生団体間の文化差で他大学展開が難航 | 低（フェーズ 2 以降） | 起業系サークルから順に、横展開時にヒアリングを行う |
| Supabase / Vercel 無料枠超過 | 低 | アクティブ数を監視、超過時は有償化 or 制限機能の追加 |
