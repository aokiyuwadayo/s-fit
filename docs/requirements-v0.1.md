## 1. このアプリは何か

**起業部のメンバーが「言いづらいことを言える場」と「一人で行きづらいイベントに一緒に行ける場」を持つための、部の公式 Web アプリ。**

- **誰の**: 起業部の現役メンバー、および運営チーム（将来的には OB および他大学サークルにも開放可能性あり）
- **何の**: 匿名で改善要望・アイデアを出せる「意見箱」と、イベント参加の同行者を募れる「掲示板」を中核機能とする内部ツール
- **なぜ**: 起業部の理念「結果重視・居場所重視」（[`MEMORY: kigyou-bu-philosophy-design`]）を実装上で支えるため。意見箱は「結果（部の改善サイクル）」を、掲示板は「居場所（行きやすさ）」を担う

## 2. MVP スコープ

### やる
- 招待コード + Google ログインによるメンバー認証
- 匿名意見箱（ハッシュ化匿名 + モデレーションキュー方式）
- イベント掲示板（投稿・閲覧・「行きたい」表明）
- 運営向けモデレーション画面
- メンバー名簿（最小限：表示名・加入日）

### やらない（v0.1 では）
- DM・チャット機能（Slack/LINE で十分）
- カレンダー連携（手動でイベント情報を貼る）
- 通知（プッシュ/メール）— v0.2 で検討
- OB 専用機能
- 他大学サークル切り替え機能（団体 ID は埋めておくが UI なし）
- ファイルアップロード（画像含む）— 初期はテキストのみ
- 検索機能
- 投稿への返信スレッド（v0.2 で検討）

## 3. ユーザーストーリー

### 部員として
1. 部に新しく入ったとき、運営から受け取った招待リンクをクリックして Google アカウントでサインインし、すぐに利用を開始できる
2. 部の運営や活動について「言いづらい意見」を匿名で投稿でき、自分が特定されない安心感を持てる
3. 同じ匿名意見に対して「自分も思っていた」と共感を示せる（リアクション）
4. 行きたい外部イベント（勉強会・ピッチコンテスト等）を投稿して、一緒に行く人を募れる
5. 他の部員の投稿したイベントに「行く」と表明でき、誰が行くかを見てから自分も行くか決められる
6. 自分の投稿（記名分）を一覧で確認・削除できる

### 運営として
7. 招待コードを発行し、誰が招待コード経由で入ったかを把握できる（ハッシュ化前の生情報を見られるのは招待コード使用時のみ）
8. 匿名意見はモデレーションキューに入り、公開前に承認/却下/編集要請ができる
9. 同一ハッシュからの違反投稿が続いた場合、そのハッシュを BAN できる（その人が誰かは特定できないが、二度と投稿させない）
10. メンバー名簿を確認し、卒業・脱退時に招待を取り消せる

## 6. 認証フロー

### 新規メンバーの加入
```
[運営] invite_codes に新しいコード発行（管理画面 or CLI）
         ↓
[運営] 招待 URL（例: yuwa.club/join?code=ABCD1234）を Slack/LINE で共有
         ↓
[新メンバー] 招待 URL を開く
         ↓
[アプリ] /join ページで「Google でサインイン」ボタン表示
         ↓
[Supabase Auth] Google OAuth → auth.users にレコード作成
         ↓
[アプリ] members レコード作成
         - organization_id = (招待コードの所属団体)
         - role = 'member'
         - anonymous_hash = SHA256(member_id + org_salt) を計算して保存
         - invite_code = (使用コード)
         ↓
[アプリ] invite_codes.used_count をインクリメント
         ↓
[新メンバー] トップ画面へ
```

### ログイン（再訪問）
- Supabase Auth セッションあり → そのまま
- セッションなし → Google でサインイン → members.id で照合 → 該当なしは「招待コードが必要です」案内

### 招待コードの方針
- デフォルト max_uses = 1（個別招待）
- 新歓イベント等では max_uses = 30 等を運営が発行可能
- expires_at は必須（デフォルト 14 日）

## 7. 匿名性の仕様（ハッシュ化方式の明文化）

### 設計原則
**「運営にも投稿者は特定できない」を技術的に担保する。** 厳しいモデレーションは BAN とモデレーションキューで実現し、匿名性は破らない。

### `anonymous_hash` の生成
```
anonymous_hash = SHA256( member.id + organization.anonymity_salt )
```
- `anonymity_salt` は organization ごとに 1 つ、サーバー側にのみ保存（環境変数 or Supabase Vault）
- member 作成時に 1 回計算して `members.anonymous_hash` に保存
- 同じメンバーは常に同じハッシュ → 「同一投稿者の連続違反」を識別可能
- 異なるメンバーは異なるハッシュ → 別人と区別可能

### 投稿時
- `posts.author_member_id` には member.id をそのまま保存
- ただし **API ハンドラレベルで** `is_anonymous = true` の投稿は、運営向け API でも `author_member_id` を返さない
- 運営の UI には `author_hash` の先頭 4 文字のみ表示（例: `a3f9…`）

### 「特定できる可能性」の現実的な範囲
完全な匿名性は技術的には難しい（タイミング・文体・ログから推測可能）。規約には以下を明記：

> 通常運用では運営も投稿者を特定できません。ただし、規約に違反する重大な投稿（脅迫・名誉毀損・違法行為等）が行われた場合、運営はサーバーログから投稿者を特定することがあります。この特定作業は運営チーム複数人の合意のもとでのみ実施されます。

### DB 上の防御深化
- Row Level Security (RLS) で `posts.author_member_id` カラムへの SELECT を運営ロールでも禁止（モデキュー用ビューでは除外）
- 監査ログ `moderation_actions` で「特定作業をしたかどうか」を必ず記録

## 8. モデレーション運用

### 誰が
- `role = 'moderator'` または `role = 'admin'` を持つメンバー
- MVP では 3-4 名（青木 + 部長 + 副部長 等を想定）
- 一人で BAN/特定ができない設計（重大アクションは複数人レビュー推奨、運用ルールで担保）

### いつ
- 新規投稿があったとき：モデレーターに通知（Slack Webhook or メール、v0.2）
- 目標 SLA: 投稿から **48 時間以内に承認/却下**
- 24 時間放置されたら別のモデレーターに自動エスカレーション（v0.2）

### どう承認
- モデレーションキュー画面で 1 件ずつ確認
- 選択肢：
  - **承認して公開**: そのまま公開
  - **編集を要請**: 投稿者に通知し、修正後に再キュー（v0.2、初期は却下のみ）
  - **却下**: 公開しない。理由を記録（投稿者には通知しない）
  - **BAN**: そのハッシュを `banned_hashes` に追加。以後そのハッシュからの投稿はキューに入る前に拒否

### モデレーションの判断基準（v0.1 ドラフト）
- ✅ 公開: 部や運営への建設的な批判、活動アイデア、改善提案
- ⚠️ 編集要請: 主旨は OK だが個人攻撃を含む、事実誤認を含む
- ❌ 却下: 個人への誹謗中傷、ハラスメント、関係ない宣伝、スパム
- 🚫 BAN: 上記の繰り返し、脅迫、違法行為の示唆

## 9. 利用規約ドラフト（箇条書き）

> 弁護士監修ではない暫定版。本番運用前にレビュー必須。

1. **対象**: 起業部メンバー（招待コード経由で加入した者）に限る
2. **禁止事項**:
   - 個人への誹謗中傷、ハラスメント、差別的発言
   - 個人情報の無断掲載（他メンバー・部外者を問わず）
   - 商業目的の宣伝（部の活動と無関係なもの）
   - 違法行為の示唆・幇助
   - スパム、システムへの攻撃行為
3. **匿名性**:
   - 匿名投稿の投稿者は通常、運営にも開示されない
   - ただし上記禁止事項のうち重大なもの（脅迫・名誉毀損・違法行為）に該当する投稿は、運営がサーバーログから特定する場合がある
   - 特定作業は運営チーム複数人の合意により実施
4. **個人情報の取り扱い**:
   - 保存項目: Google アカウントのメールアドレス、表示名、加入日、投稿内容
   - 第三者提供: 行わない
   - 退会時: メンバーレコードを `status = left` にする（投稿は匿名化して残す）
   - 完全削除を希望する場合は運営に申請
5. **モデレーション**:
   - 全ての意見投稿は運営の承認後に公開される
   - 却下理由は投稿者には開示しない場合がある
6. **アカウント停止**:
   - 規約違反が続いた場合、運営は予告なくアカウントを停止できる
7. **規約の変更**:
   - 規約変更時は Slack または YUWA 内お知らせで通知する
8. **準拠法**: 日本法
